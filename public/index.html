Overview

We built this starter kit to solve the repetitive setup of Rust web services. Instead of wiring up authentication, database pools, and error handling from scratch every time, Axum Starter provides a solid, opinionated foundation that scales from side-project to production.

Key Features

⚡️

Axum 0.8

Built on the latest version of the ergonomic and modular web framework from the Tokio team.

🗄️

Diesel ORM

Type-safe database interactions supporting SQLite (dev) and PostgreSQL (prod) out of the box.

🔐

JWT Auth

Secure authentication with access/refresh token rotation and argon2 password hashing.

🏗️

Clean Architecture

Strict separation of concerns: Controller → Service → Repository layers.

📑

Swagger UI

Auto-generated OpenAPI documentation using utoipa, accessible at /spec.

📂

File Uploads

Built-in support for multipart file uploads with streaming and validation.

Project Structure

The project follows a modular, vertical-slice architecture where features are grouped together.

  • 📂 src/modules/ - Feature modules (auth, user, attachment)
  • 📂 src/models/ - Shared domain models
  • 📂 src/extractors/ - Reusable Axum extractors (AuthUser)
  • 📂 src/utils/ - Shared utilities (error handling, JWT, encryption)
  • 📂 src/docs/ - OpenAPI documentation definitions
  • 📂 guide/ - Detailed project documentation
src/
├── main.rs              # Entry point
├── lib.rs               # Module declarations
├── modules/             # Feature modules
│   ├── auth/            # Authentication logic
│   ├── user/            # User management
│   └── attachment/      # File uploads
├── extractors/          # Custom Axum extractors
├── docs/                # OpenAPI docs
└── utils/               # Shared utilities

Conventions & Architecture

// 1. Repository (DB access)
pub async fn find_by_id(db: &DB, id: String) -> anyhow::Result {
    // Raw Diesel query
}

// 2. Service (Business logic)
pub async fn get_user(db: &DB, id: String) -> anyhow::Result {
    repository::find_by_id(db, id).await
}

// 3. Controller (HTTP)
pub async fn handler(...) -> Result {
    service::get_user(...)
        .await
        .map_err(HttpError::from_service_error)?
}

Layered Responsibility

We strictly separate concerns to ensure maintainability. The Repository handles raw data access, the Service contains business rules and validation, and the Controller manages HTTP transport.

Error Handling

Errors bubble up as anyhow::Result through the service layer. At the controller boundary, they are mapped to structured HttpError responses using a centralized mapper, ensuring consistent API error formats.
